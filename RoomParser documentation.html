Room file format:

It's a bitmap image (.bmp). Resolution is floor tiles * 2 + 1

So, for a 10x15 room, the resolution is 21x31

A bitmap is a grid of pixels, (think like Excel or something, a 2d grid of integers). It can be represented visually in an image editor/viewer, but you are actually editing a 2-dimensional grid of data values of fixed length.

What can be stored in each pixel:

Every pixel in a bitmap image has 3 channels, Red, Green, and Blue (RGB).

Each channel stores a 1-byte value indicating brightness (0x00 or 0=dark, 0xFF or 255=light).

Each pixel is represented by 3 bytes/3 channels

(000000=black, FF7733=lots of red, not so much green, very little blue, FFFFFF=white)

For the middle one, if I were to convert that to decimal, I'd say:

FF7733 = (255, 119, 51)

So, you'll see me say something like 'check the green channel' and in that case it would be 0x77 or 119 depending on context.

What is actually stored in each pixel (this is what the parser looks for):

Floor: Every pixel on coordinates that are odd in x and y (2x+1, 2x+1) is a floor tile.

Floor tiles are parsed according to the following rules:

1.check blue channel for an object type

0x00=nothing

0x01=robot's starting position

0x02=charging station

0x03=robot starting at charging station here

2.check red channel for a carpet/obstacle type:

0xFF = bare floor

0xCC = Low-Pile carpet

0x99 = High-Pile carpet

0x55 = Stairs

0x00 = Obstacle

3. assuming no obstacle, check green channel for dirtLevel:

0x00-0x64 = standard dirt level 0-100

0x65-0xFF = randomized dirt level (not yet implemented)

Example:

FF0B00 = Bare Floor, 11 dirt, no objects

Walls:Every pixel between two floor tiles (either x or y coordinate is odd) is a wall.

For now, only the red channel is checked:

0x00=Wall

0x01-0x54=doors of changing status. Will flip from open to closed and back every N times the step() function is called on the room, with N being the number 0x01-0x54

0x55=Closed Door

0xAA=Open Door

0xFF=Open

Note that there are some pixels left open (one out of every four, the wall corners). For now, these pixels aren't being checked.

How to make room files (images):

Make an image that matches samplefloor.bmp or PathingStressTest.

It helps to have:

A nice image editor like Photoshop, GIMP, or Paint.net (Microsoft Paint will work but is kind of annoying)

A hex editor like HxD or xvi32 that can show you the raw hex data (so you know what the RoomParser class will see)

  1. Set up your colors (shades of red for floor, shades of grey for walls) based on the stuff in the previous section of this guide
  2. Set up image resolution/size
  3. Set up layers/plan it out. Starting from the top left, every even row and column is a wall (including the 0th and last), everything that is odd in both row and column is a floor tile. If you have a nice editor like Photoshop, you can use layers/masks to split this up automatically (so later, you just use a single stroke of the pencil tool at a large size to cover the whole thing in black every other pixel or in red every other pixel.
  4. Actually draw in the rooms and stuff!
  1. It helps to do each channel separate, especially if you have a nice editor. GIMP, for example, has a 'lighten only' option on the pencil tool so you can leave the green and blue channels completely dark (e.g. FF0000) and then use a brush at (001100) and it will add. You'll end up with (FF1100) which is a bare floor tile with 17 dirt (remember to convert from Hexadecimal to Decimal).
  1. Save as 24-bit BMP without compression. If asked, the format is R8G8B8

What we can do in later iterations:

Obviously we can handle more types of objects and even more detailed objects but if we do, we subtract from the human-readability. This format can also be extended by increasing the resolution (using 3 pixels per tile is conservative, by scaling the image up we could easily support paths, animations, etc.)

Overall I did this because it's actually easier to use than a markup language or json. It's visual and human-editable by hand (as opposed to some type of awful tree structure/grid represented by a text file that, while technically editable, would be a real pain). With that in mind, we could also write a simple GUI editor that exports to this format as well.

We just have to keep track of these rules:

  1. Magic numbers that are used for specific things can never go past RoomParser. If the Room has to remember that High-Pile Carpet=0x99, we've failed completely. These are just part of the image format (which could be replaced by some other format). Ideally we wouldn't need any magic numbers at all, but arguably "low-pile carpet" is just as annoying to memorize and with proper documentation/image editor setup it shouldn't be an issue.
  2. Everything that's implemented in the bitmap file must be implemented in the Room class (like, there's no point to putting a value for charging stations in the RoomParser because we don't have that in the project yet)
  3. Everything that's implemented in the Room class must be implemented in the bitmap file (there should be one file that uniquely and accurately represents every possible Room object, otherwise there's functionality we have available that we're not using in practice)

How the parser actually parses:

The bitmap format is super simple if you're not using compression or anything. (http://www.dragonwins.com/domains/getteched/bmp/bmpfileformat.htm)

There's a 14-byte file header, followed by a 40-byte image header, followed by the raw pixel data. There's probably a library that does this natively, but we don't really need it. It's a bit special snowflake (coding it myself) but I was staring at it like 'i know the exact amount of data I want, and how to do it'. Sure enough, this whole thing took me maybe two hours.

Here is the header with variables (in hex):

42 4D SS SS SS SS 00 00 00 00 36 00 00 00

28 00 00 00 XX XX XX XX YY YY YY YY 01 00 18 00 00 00 00 00 40 05 00 00 13 0B 00 00 13 0B 00 00 00 00 00 00 00 00 00 00

All integers are little-endian (least significant byte first)

S = 4 bytes, the total size of the file (including headers), in bytes

X = width in pixels

Y = height in pixels

Example Header:

42 4D 76 05 00 00 00 00 00 00 36 00 00 00

28 00 00 00 15 00 00 00 15 00 00 00 01 00 18 00 00 00 00 00 40 05 00 00 13 0B 00 00 13 0B 00 00 00 00 00 00 00 00 00 00

This has a total size of 1398 bytes, and is 21x21

Everything else is pixel data. pixel data goes from bottom to top, then left to right.

Each row is padded to be a multiple of four bytes (my parser still reads pixel by pixel).

That stuff is explained on the first page of this thing.
